# Project Rules & Multi-Agent Workflow Initialization

This skill provides a standardized, repeatable procedure for setting up a multi-agent collaboration environment in a target project repository. It enforces a structural baseline and ensures that AI rules are centralized, thus eliminating context fragmentation between different agent platforms (Antigravity/Gemini CLI vs. Claude Code).

## Core Philosophy

1. **Single Source of Truth**: The central rule store must reside exclusively at `.progress/common-rules.md`. 
2. **Rule Mirroring**: All platform-specific rule files (like `rules.md` and `CLAUDE.md`) in the project root must seamlessly mirror `.progress/common-rules.md` to prevent multi-agent divergence.
3. **Structured Context Storage**: Segregate domain intelligence to prevent the root directory from becoming unwieldy.

## The Initialization Procedure

When tasked with initializing project rules or the multi-agent workflow, execute the following steps precisely in order.

### Step 1: Establish the Progress Directory Architecture

Create a `.progress` folder at the root of the project to encapsulate all AI-agent generated documentation and workflow trackers.

Run the following directory creation commands (or use robust equivalents):
`mkdir -p .progress/dev-docs .progress/tasks .progress/review-docs .progress/rewind-docs`

Then populate `.gitkeep` files in empty directories to ensure they are tracked by version control:
- Create `.progress/review-docs/.gitkeep`
- Create `.progress/rewind-docs/.gitkeep`

**Directory Purposes:**
- `dev-docs/`: For architectural, workflow, and overarching plan blueprints.
- `tasks/`: For tracking execution state. Must contain `todo.md` (active work items) and `enh-todo.md` (non-blocking enhancements, debt, and reviewed items).
- `review-docs/`: Destination for reports exclusively generated by Review Agents.
- `rewind-docs/`: Archive location for retrospective post-mortems of extremely difficult bugs.

### Step 2: Establish the Single Source of Truth

Synthesize the project's technical architecture, baseline instructions, React/Next.js performance optimization practices, and code review methodology into a single unified Markdown file.

Create **`.progress/common-rules.md`** with the following baseline structures (adapt specific technologies based on the project at hand):

1. **Core Interaction Protocol**: Language preferences (e.g., Simplified Chinese for output), "think before you act" requirements, file mutation limits (max 3 files before splitting task), and continuous rule evolution.
2. **Collaboration Setup**: Formal explanation of `.progress` folders (`tasks`, `review-docs`, `rewind-docs`) and inter-agent operability boundaries.
3. **Architecture & Standards**: Framework stack configuration, architectural data-flow, component splitting rules, rendering optimization mandates, etc.
4. **Code Quality & Code Review Reception**: Strict parameters against token superficiality—telling the agent to verify before implementing, rejecting "You're absolutely right" sycophancy, pushing back against YAGNI violations, and enforcing atomic commit verifications.

*Crucial Directive: Make it expressly clear within this file that IT IS the only source of truth. Any other AI instructions flow outwards from this core.*

### Step 3: Mirror to Platform-Specific Rule Files

Platform-specific endpoints (Gemini/Antigravity CLI -> `rules.md`, Claude Code -> `CLAUDE.md`) require corresponding files in the project root.

To prevent divergence, inject a preamble into these specific files linking them permanently back to `.progress/common-rules.md`, followed by an absolute verbatim copy/mirror of the text inside the common rules.

#### Create/Overwrite `rules.md` (For Antigravity/Gemini CLI)
Add the preamble:
```markdown
# 🚦 [Project Name] (Antigravity / Gemini CLI) Instruction Specification

> **⚠️ Sync Declaration**
> This file is maintained to ensure consistent distribution across multiple agents. The authoritative source and single source of truth for this specification is the project's common rule file: `.progress/common-rules.md`.
> As a primary development and assistant agent within this project, all your judgments and low-level operations are governed by the **common rules** below. If the project specification changes or needs to evolve, you must immediately update `.progress/common-rules.md` first!

---
```
*(Append the full content of `.progress/common-rules.md` immediately following this preamble.)*

#### Create/Overwrite `CLAUDE.md` (For Claude Code)
Add the preamble:
```markdown
# 🚦 [Project Name] (Claude Code) Code Review & Collaboration Specification

> **⚠️ Sync Declaration**
> This file is maintained to enforce constraints across LLM-based multi-platform clients (such as Claude). The authoritative source and single source of truth for this specification is: `.progress/common-rules.md`.
> As a code review or collaborative development agent, all your verification, review, and commit work is governed by the **common rules** listed below. Whenever your understanding of the project is refined or you receive a human instruction that changes behavior, reflect the change in `.progress/common-rules.md` first, then propagate it to client-specific files like this one.

---
```
*(Append the full content of `.progress/common-rules.md` immediately following this preamble.)*

### Step 4: Validate Implementation

After creating the directory structure and the files, verify:
1. Does `.progress/common-rules.md` explicitly absorb the project architecture intent?
2. Does `rules.md` contain the preamble and the mirrored common rules?
3. Does `CLAUDE.md` contain the preamble and the mirrored common rules?
4. Are `review-docs/` and `rewind-docs/` successfully generated with `.gitkeep`?

Conclude by informing the user that the synchronization was successful, reinforcing that any future global behavioral changes should exclusively target `.progress/common-rules.md`.
